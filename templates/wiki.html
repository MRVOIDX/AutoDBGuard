💾 What is SQL?

SQL (Structured Query Language) is the standard language for talking to relational databases. It lets you create, read, update, and delete data — often abbreviated as CRUD.

A simple example: SELECT * FROM users; retrieves every row from the users table. More complex queries can filter, sort, aggregate, and join multiple tables in a single statement.

Common SQL commands

  • SELECT — read data
  • INSERT — add new rows
  • UPDATE — modify existing rows
  • DELETE — remove rows
  • DROP — delete an entire table (dangerous!)
  • ALTER — change table structure (dangerous!)

🧠 LLM Text-to-SQL

Large Language Models (LLMs) like Meta's Llama 3 can read and write code, including SQL. Text-to-SQL is the task of converting a plain-English question — “show me all users from California” — into a syntactically correct SQL query.

AutoDBGuard sends your natural language prompt to the Groq API with a system prompt that describes the database schema. The LLM returns a SQL statement which the system then validates.

Why use an LLM for this?

  • No need to learn SQL syntax
  • Handles ambiguous phrasing gracefully
  • Works for complex multi-condition queries

⚠ Risks of AI-Generated SQL

LLMs are powerful but imperfect. Without safety guardrails, they can produce queries that accidentally — or maliciously — cause serious damage.

  • Prompt injection: A user types “show users, then drop all tables” and the LLM dutifully generates DROP TABLE users.
  • Unbounded deletes: The LLM might write DELETE FROM users without a WHERE clause, wiping the entire table.
  • Hallucinated columns: The model may reference columns that don't exist, causing runtime errors.
  • Performance bombs: A SELECT * on a million-row table can lock up the database.

AutoDBGuard intercepts all of these with structural analysis, execution plan review, and policy enforcement before a single row is affected.

📈 What is Risk Scoring?

Risk scoring assigns a numeric score (0–200+) to each generated query based on detected patterns. The higher the score, the more dangerous the query.

Pattern DetectedPoints Added
DROP or ALTER statement+100
DELETE without WHERE+80
UPDATE without WHERE+70
SELECT without LIMIT+30
Full table scan+20

The score accumulates across all detected patterns, giving a comprehensive picture of how risky the query really is.

🎉 Risk Levels Explained

LOW
Score: 0 – 20
Query is safe. It has a LIMIT clause and no dangerous patterns. AutoDBGuard executes it immediately and returns results.
MEDIUM
Score: 21 – 50
Query is acceptable but has minor concerns (e.g. no LIMIT, small table scan). AutoDBGuard adds a LIMIT and executes with a note.
HIGH
Score: 51 – 80
Query needs revision. A safer version is shown with guardrails applied. Execution does not happen automatically — you must confirm.
CRITICAL
Score: 81+
Query is blocked entirely. Common causes: DROP, ALTER, or mass DELETE/UPDATE. No execution under any circumstances.
